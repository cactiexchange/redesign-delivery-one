1 · Executive summary

The management flow (screenshot, steps 1–4) and the current build differ in a predictable way: step 3 — the onsite deploy engine — is largely done; the gaps cluster at step 1 (pre-engagement documentation), step 4 (as-built automation), and the session/credential model. This plan closes every gap with the G1–G7 + G9 workstreams sequenced into four sprints, folded into the existing three-seam integration plan. Nothing already built gets thrown away; every workstream lands in one of the three repos this workspace already pins. Since v1, the G9 groundwork has shipped: researched facilities data for all 26 models and the product onboarding guide.

4 · Workstream designs

G1 — Check-in / check-out session model DONE · deploy 750b836

(v2: revised from "true offline" per Drew — online check-in and check-out are fine; only the middle must work with zero internet.) The engagement becomes a three-phase session: check-in (online: pull tenant intent + DAG from NetBox, validate, cache locally as the session bundle), deploy (air-gapped: everything runs from the cached bundle — laptop, db9/RJ45 console, ethernet, nothing else), and check-out (online again: export the as-built package — credentials JSON, validation results, captured artifacts — and push completion status back to NetBox, plus the optional 1Password sync from G3).

  • check-in subcommand: runs the existing FetchNetboxData() + full DAG validation, then seals the resolved DeploymentPlan into session_bundle_<tenant>.enc — same AES-GCM + HMAC pattern as hardware_matrix.enc, keyed to the build. Can run at the office or over any connection before going dark.
  • Deploy phase: a present bundle is the default source — no network attempted. Live NetBox becomes the explicit --online opt-in. Staleness (per D4): warn at 7 days, block at 30 with an override prompt.
  • check-out subcommand: when connectivity returns — packages the as-built export (credentials JSON, validation results, artifact manifest) into one directory for reparsers-v2 (G6), pushes per-device deployment status back to the NetBox tenant, and offers the 1Password sync (G3). Check-out is re-runnable; nothing is lost if it fails midway.
  • Retires open finding RT-2: the compiled-in NetBox fallback token is deleted in the same commit. The hotspot narrative leaves the docs.
  • Acceptance: check-in → NIC disabled → full mock deployment → NIC restored → check-out; unit test proving zero network dials during the deploy phase; docs updated.

G3 — Randomized passwords + pluggable credential backends gap deploy · M

(v2: expanded per Drew — local vault is the default backend, 1Password is an option for every password-touching step, and all new passwords are randomized and unique per device/service.)

  • Password generation becomes the default. Everywhere the tool currently asks the engineer to invent a password (ESXi root, OneFS root + admin, PowerStore admin, switch admin…), it instead generates a cryptographically random one — unique per device and per service, length/charset tuned per product's documented rules (e.g. PowerStore complexity requirements, OneFS length limits). The engineer sees it only at the prompt moment and at check-out; nothing is ever reused across devices. A --manual-passwords escape hatch stays for customer-dictated values.
  • Backend interface over the credential store: VaultBackend with two implementations — local (today's AES-GCM file, the default, fully offline) and 1password (via the op CLI). Backend selection at check-in; the 1Password backend requires connectivity, so onsite it queues writes locally and flushes them at check-out (G1) — the engineer never blocks on it mid-deployment.
  • 1Password item layout: one item per device, title <tenant>/<device>, fields for factory + final credentials, OS password, final IP, serial, validation status, tagged by tenant. After a verified flush, the tool offers to delete the plaintext as-built JSON.

Acceptance: generated passwords pass each product's complexity rules (unit-tested per product); no two devices in a tenant share any password (asserted in tests); mock op binary exercises the queue-and-flush path; documented in admin + engineer guides.

G4 — Full step control gap deploy · S–M

(v2: expanded per Drew — beyond interrupt/resume, parts must be skippable, re-runnable, and easy to go back to.) Interrupt/resume already works (per-device state file). This adds the rest of the control surface:

  • done (human takeover) at the failure prompt: "I completed this device by hand" — marks StateDone with manual: true, prompts for the final credentials/IP so the vault and as-built stay truthful. Phase 5 still validates it like any other device — a hand-built switch that fails VLT checks still turns the run red.
  • redo (go back): re-run any completed or skipped device. Phase-order guard applies in reverse — redoing a switch warns that dependent storage/compute may need revalidation, and flags their Phase 5 results stale.
  • Device picker: instead of only marching forward, the run menu lists every device with its state (pending / in-progress / done / done-manual / failed / skipped) and lets the engineer jump to any of them — the DAG still blocks selections whose phase prerequisites aren't met.

G6 — As-built schema + reparsers-v2 import (Seam 3) gap reparsers + deploy · M

(v2: confirmed with Drew — yes, this is reparsers-v2, the as-built document generator.) Closes the loop on slide step 4. First task is archaeology: read reparsers-v2's document builder and template internals to map what it consumes today. Then:

  • Schema v1 (docs/asbuilt_schema.md in the umbrella repo): versioned JSON — tenant, engineer, dates; per device: model, serial, final IPs, credentials, validation results, and an artifact manifest pointing at the Seam 2/G5 captures. The deploy tool's existing export is ~80% of this already; it gains the manifest and a schema_version field.
  • reparsers-v2 importer: new input type that ingests the JSON + artifact directory and emits the device inventory, credential appendix, and validation summary sections of the as-built; the diagrammer's draw.io/PNG export drops in as the topology figure.
  • Result: engineer returns from site with one directory; the as-built is one command. "The program spits out the as builts" becomes literally true.

G9 — Product onboarding framework guide shipped · registry pending deploy · M

(v2: promoted from a positioning note to a real workstream per Drew — "a large focus, plan how we onboard new equipment from any manufacturer," including cards/modules.)

Already landed (2026-06-11):

  • docs/product_onboarding.md — the three-tier onboarding model: Tier 1 new model in a known family = pure JSON data (~30 min, no compile); Tier 2 new NIC/card/module = two JSON edits (~10 min, un-onboarded cards are blocked at intake by the DAG — by design); Tier 3 new product line (incl. non-Dell: Cisco, HPE, Pure…) = one new Go deploy-flow file plus registry entries, with a 13-step checklist mapping every file that must be touched.
  • Facilities research workflow proven on all 26 existing models (G2) — the same spec-sheet method documented in docs/facilities_sources.md applies to any manufacturer.

Remaining build item — the product registry refactor: today, wiring a product line touches five Go files (fetch validation, phase map, deploy dispatch, validator, artifact set). Before the second non-Dell line lands, dispatch centralizes into one product_registry.go table where each line declares its matcher, phase, required fields, factory creds, deploy fn, validator, and artifact set — so the compiler enforces complete onboarding, and a CI rule asserts every seeder category has a registry entry. After that: new line = one file + one registry literal.

Honest scaling cost (for management conversations): tiers 1–2 are genuinely free; tier 3's deploy flow is real protocol work — budget what one Dell line took (~1–2 days build plus field-test fixes), and a line isn't "done" until it survives one field test.

7 · Risks

  • Field test №1 is the real gate. Every Phase 4/5 flow plus G1/G4/G5 are written against documented device behavior; first hardware contact will find surprises. The sprint plan deliberately front-loads everything the field test can exercise and keeps sprints 3–4 software-only.
  • G2 data quality: power/RU numbers come from Dell spec sheets — same provenance and maintenance burden as the existing hardware matrix. Mitigation: the facilities block is per-model and versioned with the matrix seal.
  • G6 unknowns: reparsers-v2 internals haven't been read yet; the 2–4d estimate hardens after a half-day archaeology pass (first task of sprint 3, can be pulled earlier).
  • Solo throughput: ~4.5 weeks of plan at solo pace. If the meeting creates urgency, sprints 2 and 3 parallelize cleanly across two people (different repos, defined interfaces).
