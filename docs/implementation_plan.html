1 · Executive summary

The management flow (screenshot, steps 1–4) and the current build differ in a predictable way: step 3 — the onsite deploy engine — is largely done; the gaps cluster at step 1 (pre-engagement documentation), step 4 (as-built automation), and three specific claims the current architecture contradicts or doesn't yet back (no-internet operation, any-manufacturer scope, 1Password). This plan closes every gap with eight workstreams (G1–G8) sequenced into four sprints, folded into the existing three-seam integration plan. Nothing already built gets thrown away; every workstream lands in one of the three repos this workspace already pins.

4 · Workstream designs

G1 — Offline deployment bundle gap deploy · M

Split the engagement into an online office phase and an air-gapped site phase, matching slide step 3 exactly ("pull the DAG instructions and workflow from the source of truth online and then use that to deploy").

  • New subcommand snapshot: runs the existing FetchNetboxData() + full DAG validation, then seals the resolved DeploymentPlan (devices, contexts, storage architecture, hardware matrix checks pre-passed) into site_bundle_<tenant>.enc — same AES-GCM + HMAC pattern as hardware_matrix.enc, keyed to the build.
  • Deploy mode: if a bundle sits next to the exe, it is the default source; live NetBox becomes the explicit --online opt-in. Stale-bundle guard: warn if older than 7 days.
  • Retires open finding RT-2: the compiled-in NetBox fallback token is deleted in the same commit. The hotspot narrative leaves the docs.
  • Acceptance: full mock deployment with the NIC disabled; unit test proving zero network dials in offline mode; docs updated.

G2 — PEQ documentation generator gap deploy (python) · L

One command, run right after seeding: python peq_generator.py --tenant <slug> → a customer-facing Pre-Engagement Questionnaire/Requirements doc (HTML, printable) covering every bullet in slide step 1c. The trick that makes this cheap: almost everything is already derivable — the only net-new data is a per-model facilities table.

G3 — 1Password sync gap deploy · S

Decision needed (see §6): the honest version of this promise is a post-engagement sync at the office, not live-onsite — 1Password needs connectivity, the site doesn't have any (G1's whole point). Proposed:

  • orchestrator sync-1password reads the encrypted vault (credentials_vault_<tenant>.json.enc) and creates/updates one item per device via the op CLI (already authenticated on the office workstation): title <tenant>/<device>, fields for username, password, OS password, final IP, serial, validation status; tagged with the tenant slug.
  • After a verified sync, the tool offers to delete the plaintext as-built JSON — strengthening the existing "merge and delete" rule rather than weakening it.
  • Stays out of the field exe path entirely; zero impact on the air-gap story.

Acceptance: mock op binary in tests; manual verification against a real vault once; documented in the admin guide.

G4 — Human takeover ("done") gap deploy · S

At the failure prompt, add a third option alongside retry/skip: done — "I completed this device by hand." Marks the device StateDone with a manual: true annotation, prompts for the final credentials/IP so the vault and as-built stay truthful, and Phase 5 still validates it like any other device — a hand-built switch that fails VLT checks still turns the run red. Smallest item in the plan, directly covers slide 3c.i.

G6 — As-built schema + reparsers-v2 import (Seam 3) gap reparsers + deploy · M

Closes the loop on slide step 4. First task is archaeology: read reparsers-v2's document builder and template internals to map what it consumes today. Then:

  • Schema v1 (docs/asbuilt_schema.md in the umbrella repo): versioned JSON — tenant, engineer, dates; per device: model, serial, final IPs, credentials, validation results, and an artifact manifest pointing at the Seam 2/G5 captures. The deploy tool's existing export is ~80% of this already; it gains the manifest and a schema_version field.
  • reparsers-v2 importer: new input type that ingests the JSON + artifact directory and emits the device inventory, credential appendix, and validation summary sections of the as-built; the diagrammer's draw.io/PNG export drops in as the topology figure.
  • Result: engineer returns from site with one directory; the as-built is one command. "The program spits out the as builts" becomes literally true.

G7 — Wizard → seeder export (Seam 1) gap diagrammer + deploy · M

Makes the diagrammer the front door management saw in step 1, without letting it bypass validation:

  • Diagrammer: "Export deployment intent" button on the Review step — emits project_intent.json mirroring the workbook's logical schema (project, storage architecture, per-product model/count/values). Trivial to produce from wizard state.
  • Seeder: --from-json project_intent.json — reuses the exact validation/dry-run/push path the workbook uses (the loader logic already exists; this is a second front-end onto it). The DAG checks stay mandatory regardless of entry point.
  • Workbook intake remains for engagements that start in Excel; both converge on the same pipeline.

G8 — BOM / Jira prefill deferred diagrammer · M

Parse a Dell BOM/quote export to prefill wizard models and counts (Jira via CSV export of the engagement ticket). Deferred until we have a real BOM sample to parse — guessing the format is wasted work. Slot: sprint 4 or whenever a sample lands. The wizard/workbook remain the manual path meanwhile, so nothing blocks on this.

5 · Sequencing — how this folds into the current plan

The three-seam plan stays intact; the gaps slot around it. G1 jumps to the front because it's the only item where the slide and the architecture actively disagree, and it deletes a standing security finding. The first hardware field test gates between sprints 2 and 3 — it exercises everything sprint 1–2 produce.

What to demo at next week's meeting (achievable: sprint 1 + existing build): end-to-end on the mock tenant — workbook → NetBox → snapshot → pull the network cable → offline deploy with halt/resume and a human-takeover → validation → diagrammer ingesting the captured zip. That covers slide steps 2 and 3 completely, demonstrates the air-gap claim for real, and shows step 4's raw materials. Steps 1 and 4 are presented as this plan (sprints 2–3).

7 · Risks

  • Field test №1 is the real gate. Every Phase 4/5 flow plus G1/G4/G5 are written against documented device behavior; first hardware contact will find surprises. The sprint plan deliberately front-loads everything the field test can exercise and keeps sprints 3–4 software-only.
  • G2 data quality: power/RU numbers come from Dell spec sheets — same provenance and maintenance burden as the existing hardware matrix. Mitigation: the facilities block is per-model and versioned with the matrix seal.
  • G6 unknowns: reparsers-v2 internals haven't been read yet; the 2–4d estimate hardens after a half-day archaeology pass (first task of sprint 3, can be pulled earlier).
  • Solo throughput: ~4.5 weeks of plan at solo pace. If the meeting creates urgency, sprints 2 and 3 parallelize cleanly across two people (different repos, defined interfaces).
