Field Engineer Guide

For the L2 engineer running the deployment on site. This is the friendly, consolidated version; for the first hardware run also use the step-by-step field test runbook. Companions: System Admin · DevOps Admin.

The golden rule: the tool never fakes success. If a device doesn't answer, it stops and shows you the transcript or API response. Never work around a FATAL DAG ERROR or a red validation to "keep moving" — fix it or record it and raise it.

The shape of a job: three phases

You are briefly online at the start and end, and fully air-gapped in between. No hotspot is needed on the datacenter floor.

What to bring (the deployment kit)

  • Your minted datacenter_orchestrator_<You>.exe — check the expiry covers the trip.
  • hardware_matrix.enc in the same folder as the exe.
  • session_bundle_<project>.enc from your check-in (it only opens in your build).
  • A stock ESXi installer ISO if the job has PowerEdge nodes (download before you travel).
  • Console cables: RJ45-to-DB9 (switches), USB-RS232 (PowerScale), micro-USB (Unity SP), + an Ethernet patch cable (ME5, PowerStore discovery).
  • A USB serial adapter with known-good drivers — test it first; no serial port found is a hard stop.
  • The device pull tags (factory iDRAC/array passwords).

Phase 1 — Check-in (online, before you travel)

set NETBOX_TOKEN=<token>
deploy.exe check-in
-> Enter Target Sequence Identifier: project-<name>

It pulls the design, validates it against the hardware rules, and seals session_bundle_<project>.enc. If check-in reports errors, fix them in NetBox now — not on the floor. A bundle older than 7 days warns; older than 30 days blocks (re-run check-in).

Phase 2 — Deploy (air-gapped)

deploy.exe
  1. License: if offline, you'll see a Challenge Code. Call the NOC; they read back a 6-digit override. Enter it. (This is normal on networks where the time-check can't reach out.)
  2. The bundle loads and re-validates offline. Zero NetBox traffic from here.
  3. Customer-provided switches are listed first with recommended port configs. We do not configure them — coordinate the recommendations with the customer's network team before deploying gear that depends on those ports.
  4. Deployment runs switches → storage → compute. For each device: cabling instructions, then the real procedure.

Passwords are generated for you. Each device's new OS password (ESXi root, PowerStore admin, OneFS root) is random, unique, and shown once. It's saved in the vault and reappears at check-out — note it or trust the export. Factory pull-tag passwords are still prompted (and reused on retries). If the customer dictates passwords, restart with --manual-passwords.
Interruptions are safe. Progress saves per device. If the run dies at device 5 of 7, rerun the exe — done devices are skipped, and the control menu lets you pick up anything pending/skipped/failed.

Validation (automatic, at the end of Phase 2)

After the last device, live checks run against everything deployed, and CLI/REST captures are saved for the as-built. Expected passes:

ProductA pass means
OS10VLT peer is up (both switches deployed + VLTi cabled)
SONiCat least one BGP neighbor Established
PowerStorecluster state = Configured
ESXiroot SSH works; active storage paths (array-primary)
PowerScaleOneFS responds (isi status)
PowerVault ME5system health OK
UnityUnisphere reachable (full health in the UI)
A red validation exits non-zero — do not leave site with validation red. The detail line tells you why (cabling, zoning, peer not deployed, paths not visible). Fix it, then redo N the device, or finish it by hand with done (it's still validated).

Phase 3 — Check-out (back online)

set NETBOX_TOKEN=<write-token>
deploy.exe check-out

The token needs write scope (a 403 means it doesn't). Check-out:

  • Builds checkout_<project>/: the plaintext credentials JSON + a manifest.json (per-device status, validation, artifact list) for the as-built generator.
  • Reports to NetBox: deployed devices flip to Active; every attempted device gets a journal entry.
  • Optional --sync-1password: pushes credentials to 1Password, then offers to delete the plaintext.
  • Re-runnable — if any write fails, fix and run again.
Before the laptop leaves your control: merge the as-built credentials into the as-built doc and delete asbuilt_credentials_<project>.json and any *_esxi_ks.iso in output_configs/ (they embed the ESXi root password). 1Password sync does the delete-offer for you.
