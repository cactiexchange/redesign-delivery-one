1 · One-time setup

pip install -r requirements.txt        # pynetbox, openpyxl
go version                              # for building the field exe (Go toolchain)

Set NetBox access for the admin scripts:

set NETBOX_URL=https://<your-netbox>
set NETBOX_TOKEN=<admin-token>
Token hygiene (RT-2): the Python admin scripts historically carried a default token. Rotate it and rely on NETBOX_TOKEN. The field binary no longer embeds any fallback.

2 · Seeding a project into NetBox

Three intake paths, all converging on one validating push (push_to_netbox). Always --dry-run first — it prints a CREATE/UPDATE/UNCHANGED diff and writes nothing.

a. Workbook (preferred for anything sizable)

python devops_netbox_seeder.py --template project.xlsx        # blank, one sheet per product line
# architect fills Model + Count + values
python devops_netbox_seeder.py --from-xlsx project.xlsx --dry-run
python devops_netbox_seeder.py --from-xlsx project.xlsx       # push

The workbook is generated from seeder_ui_dag.json, so it can never drift from what the seeder expects.

b. Diagrammer export (Seam 1)

python devops_netbox_seeder.py --from-json intent.json --dry-run
python devops_netbox_seeder.py --from-json intent.json

The diagrammer's wizard exports intent.json (models, counts, customer switches, VLT/BGP). The seeder validates it and reports any per-device deployment IPs the wizard didn't capture — fill those in the JSON or finish in the workbook.

c. Interactive (small edits)

python devops_netbox_seeder.py
Customer-provided switches: use the Customer Switch line (port count + types + recommended VLAN/MTU). They land under a distinct manufacturer + grey role, are flagged customer_provided, and the deploy tool advises on but never configures them.

Add --append to require an existing tenant (adds devices to a project mid-stream).

3 · Generating the customer requirements (PEQ)

python peq_generator.py --tenant project-<name>     # -> peq_project-<name>.html

Produces a customer-facing HTML doc: power/cooling (peak/typical W, BTU/hr, receptacle count by type), rack space (with deep-chassis warnings), switch ports (demand vs customer supply), DNS entries, and firewall flow tables. Multi-node Gen6 chassis are grouped so power isn't over-counted. Send it to the customer to provision the site before the engagement. Power is planning-grade — confirm with Dell ESSA for contract sizing.

4 · Minting an engineer binary & the matrix

Each engineer gets a uniquely compiled exe. First seal the hardware matrix, then mint:

python dag_swarm_orchestrator.py            # -> hardware_matrix.enc (plaintext shredded)
go run .\cmd\admin\admin_builder.go         # prompts: engineer name, expiry (YYYY-MM-DD)

Hand the engineer both datacenter_orchestrator_<Name>.exe and hardware_matrix.enc (same folder). Inject a tenant-scoped token at mint time, or have the engineer set NETBOX_TOKEN. Remember: check-out needs a write-scoped token (dcim.device + extras.journalentry); check-in only needs read.

Bundles are build-bound. A session bundle only opens in the exe that created it. Each engineer checks in their own projects.

5 · NOC override duty

When an engineer is offline at unlock, the exe shows a Challenge Code (e.g. Drew-483291).

go run .\cmd\admin\admin_builder.go respond     # select the build; reads back the 6-digit code

The code is HMAC-SHA256(BuildSecret, challenge). There is no master keyword. Log each override in the engagement ticket. Note: on many networks the time-check can't reach Cloudflare, so this happens on most cold starts — it's expected, not a fault.

6 · Maintaining the hardware matrix

hardware_definitions.json is the per-model source of truth: deployment limits (NICs, NVMe capacity, firmware floors, RAM, cluster bounds) and a facilities block (rack units, depth, PSU count/watts, inlet/receptacle, watts, BTU, voltage) the PEQ consumes. After any edit:

python dag_swarm_orchestrator.py                 # re-seal hardware_matrix.enc
python scripts/build_custom_device_types.py      # sync NetBox device types (u_height from facilities)
python scripts/ci_hardware_sync_checker.py       # matrix <-> docs parity
python scripts/test_workbook_intake.py           # intake regression
python scripts/test_peq_generator.py             # PEQ regression

Facilities research method and sources are in docs/facilities_sources.md.

7 · Onboarding new products (G9)

Full guide: deploy/docs/product_onboarding.md. Three tiers:

TierWhatEffort
1New model in a known familyPure JSON: add to hardware_definitions.json (+ facilities) and re-seal. No compile.
2New NIC / card / moduleAppend to the model's supported_nics + the nic_model prompt options. No compile.
3New product line (incl. non-Dell)One Go deploy-flow file + a 13-step checklist (fetch fields, phase, factory creds, deploy fn, validator, artifacts, docs, tests). Budget what one Dell line took, plus a field test.

Un-onboarded cards are blocked at intake by the DAG — that's the safety net working, not a bug.

8 · The as-built pipeline (check-out → reparsers-v2)

The engineer's check-out produces checkout_<tenant>/: a non-secret manifest.json (deployment record + artifact inventory) and the plaintext asbuilt_credentials_<tenant>.json. Both carry schema_version. These, plus any vendor TSRs, feed reparsers-v2 to produce the customer's as-built Word document.

The shared contract and the planned --asbuilt importer are documented in docs/asbuilt_schema.md. Until the importer lands (gated on the first real check-out package), merge the credential record into the as-built manually and then delete it.

Plaintext handling: the credentials JSON and any kickstart ISO embed real passwords. Merge and delete; or have the engineer flush to 1Password at check-out (--sync-1password) and delete. Never commit either.
