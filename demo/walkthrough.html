Demo Walkthrough

A presenter script for showing the deployment ecosystem before the first hardware field test. Everything here runs without datacenter hardware. Sample artifacts in this folder let you present even without running anything live.

The one honest caveat, stated up front: with no hardware on the bench, the per-device deploy and validation steps will attempt real procedures and then correctly report the devices as unreachable. That is the point — the tool never fakes success. The demo shows the orchestration machinery (intake, validation, sessions, the air-gap, passwords, step control, check-out, the customer deliverables) working end to end; a successful rack is what the field test proves. Frame it that way and the "failures" become a feature ("watch it refuse to lie").

Track A — Customer / management deliverables fully real

These are genuine artifacts a customer receives. No caveats — show them as finished products.

1Show the Pre-Engagement Requirements doc

Open sample_peq.html in a browser (or generate live: python peq_generator.py --tenant project-gotham).

step 1 of the management flow — the project now auto-produces the customer's power, rack, port, DNS, and firewall-flow requirements from the design. This box on the slide previously had nothing behind it.
"Before anyone drives to site, the customer gets exactly what to provision — including that two of these models need 200-240V and one chassis is too deep for a standard rack. The tool caught both."
2Show intake → NetBox (the dry-run diff)
python devops_netbox_seeder.py --from-json demo/sample_intent.json --dry-run

Or the workbook path: --template project.xlsx then --from-xlsx project.xlsx --dry-run.

the design becomes the source of truth, validated. The dry-run prints a CREATE/UPDATE/UNCHANGED diff and the DAG rejects impossible designs before anything is written.
"Whether the design comes from the diagrammer or a spreadsheet, it lands in NetBox through one validating gate. Try to put 1 PowerScale node where 3 are required and it stops you here, not on site."

Track B — The air-gap walkthrough orchestration real, devices absent

This is the memorable one. Have a freshly seeded tenant ready (the stale mock tenants predate the current schema). The cable-pull is the moment that lands.

3Check-in (online)
set NETBOX_TOKEN=<token>
deploy.exe check-in
-> Enter Target Sequence Identifier: project-gotham
the engagement is sealed into one encrypted bundle. Internet is needed now and never again until check-out.
"This is the only time the laptop touches the internet on the way in. It pulls the design, validates it against the hardware rules, and seals it."
4Pull the network cable. Physically. In front of the room.
everything from here is air-gapped — the literal promise to upper management.
"From here we're the L2 engineer on a datacenter floor with no internet. Watch what still works."
5Run the offline deploy
deploy.exe

Walk through what appears: the bundle loads and re-validates offline; customer-provided switches are listed first with recommended port configs (we don't configure them); deployment proceeds in enforced order (network → storage → compute); each device generates a unique random password shown once; the per-device prompt offers y / N / done; the control menu allows skip, redo, and hand-takeover.

the full orchestration runs with zero network: sessions, phase ordering, password generation, step control, customer-switch handling. Devices report unreachable — correctly, because none are on the bench.
"It's driving the real OS10 console dialog, the real Redfish ISO install, the real PowerStore REST sequence. With no array answering, it stops and shows the transcript instead of pretending. That honesty is the whole design."
6Reconnect the cable, then check-out
set NETBOX_TOKEN=<write-token>
deploy.exe check-out

Then open sample_checkout/manifest.json to show the output shape (per-device status incl. a manual takeover, validation results, artifact inventory) and note the credentials JSON that feeds the as-built.

step 4 of the management flow — the run reports per-device status back to NetBox (Active + a journal entry) and packages the as-built materials for the document generator.
"Back online, it writes the outcome of every device back to the source of truth and hands a complete record to the as-built generator. One engagement, fully closed-loop."

Track C — Needs a machine with Node (you, not the slide deck)

The diagrammer wizard (design → export → seed) is a complete Seam 1 demo, but it needs a Node build (npm install && npm run dev in diagrammer/). Show the 6-step wizard, tick a switch as customer-provided (it renders amber/dashed), and hit Export Deployment Intent — the file it downloads is exactly the sample_intent.json shape, which then flows through Track A step 2.

Setup checklist: a minted deploy.exe + hardware_matrix.enc; a NetBox token; one freshly seeded demo tenant; this folder open in a browser; and — for the cable-pull — a laptop you can actually disconnect (mobile hotspot off too). If the license check shows a Challenge Code when offline, that's expected (Cloudflare time is unreachable) — have the NOC override ready, or pre-warm the license before the room.
